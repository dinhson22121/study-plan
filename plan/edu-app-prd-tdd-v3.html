Notification Strategy

Notification là thành phần chiến lược để tăng retention. Mỗi loại notification có trigger logic, schedule riêng và phải tôn trọng user preference.

Daily Reminder

Daily Study Reminder

Nhắc học sinh học bài hàng ngày theo giờ học đã cài trong Study Plan.

⏰ Cron theo timezone 📤 Kafka → FCM
Weekly Quiz

Weekly Quiz Reminder

Nhắc làm bài kiểm tra tuần vào tối Chủ Nhật nếu chưa hoàn thành.

⏰ Chủ nhật 19:00 📤 Kafka → FCM
Study Plan

Study Plan Reminder

Nhắc khi học sinh sắp trễ milestone trong Study Plan đã đặt.

⏰ Event-driven 📤 Kafka → FCM
Achievement

Achievement Notification

Chúc mừng khi học sinh đạt milestone, streak, hoặc hoàn thành chapter.

⚡ Event-driven 📤 Direct FCM
Re-engagement

Re-engagement Notification

Trigger sau 3 ngày không active. Nội dung cá nhân hóa theo chủ đề yếu nhất.

⏰ Batch job hàng ngày 📤 Kafka → FCM
Admin Broadcast

Admin Broadcast

Admin gửi thông báo tới tất cả hoặc nhóm học sinh (cập nhật tính năng, sự kiện).

🖱️ Manual trigger 📤 Kafka fan-out
⚠️

User Preference Gate: Trước khi enqueue bất kỳ notification nào vào Kafka, scheduler phải query notification_preference để kiểm tra enabled = true. Nếu user tắt loại đó → skip silently, log vào notification_log với status SKIPPED.

Kafka Topics

ℹ️

Mỗi message phải có correlationId để trace end-to-end từ scheduler → consumer → FCM → log. Dùng UUID v4.

notification.schedule

Producer: Study Plan scheduler, Quiz module, Progress module. Consumer: Notification Module.

JSON Schema { "correlationId": "uuid-v4", "studentId": "uuid-v4", "notificationType": "DAILY_REMINDER", "templateCode": "DAILY_REMINDER_V1", "templateVariables": { "name": "Minh", "streak": 7 }, "scheduledAt": "2026-08-10T20:00:00+07:00", "idempotencyKey": "student-123-daily-2026-08-10" }

notification.send

Producer: Notification Consumer (sau khi resolve device token). Consumer: FCM Sender.

JSON Schema { "correlationId": "uuid-v4", "deviceToken": "fcm-token-string", "title": "Đã đến giờ học rồi, Minh! 🎯", "body": "Streak 7 ngày của bạn đang chờ. Tiếp tục nào!", "data": { "deepLink": "/study/today" }, "logId": "uuid-v4" }

notification.result

Producer: FCM Sender. Consumer: Log Updater.

JSON Schema { "correlationId": "uuid-v4", "logId": "uuid-v4", "status": "SENT", // SENT | FAILED "errorCode": "TOKEN_INVALID", // null nếu SENT "shouldRetry": false, "sentAt": "2026-08-10T20:00:05+07:00" }

notification.dlq (Dead Letter Queue — bảng mới)

Các message fail sau 3 lần retry sẽ được route sang topic này để manual review.

07

Notification Flow

1
Trigger Event
Study Plan module emit StudyPlanReminderDue event. Cron scheduler (robfig/cron v3) check timezone của từng user, tạo message với idempotencyKey.
2
Preference Check
Notification Service query notification_preference. Nếu enabled = false → skip, ghi log SKIPPED. Nếu enabled → tiếp tục.
3
Enqueue Kafka
Produce message vào notification.schedule. Redis check idempotencyKey để tránh duplicate (TTL 24h).
4
Consumer: Resolve Token
Notification Consumer lấy device token từ device_token table (chỉ lấy is_active = true). Render template với variables. Create notification_log record với status PENDING.
5
Send via FCM
FCM Adapter gửi push notification. Nếu HTTP 404 (token invalid) → deactivate token, log FAILED. Nếu HTTP 429 → exponential backoff, retry tối đa 3 lần.
6
Result & Log
Produce vào notification.result. Log Updater consumer update status thành SENT hoặc FAILED. Message fail sau 3 lần → notification.dlq.

FCM Retry Strategy

Go — infrastructure/fcm_adapter.go func (a *FCMAdapter) SendWithRetry(token, title, body string, data map[string]string) error { maxRetries := 3 for attempt := 0; attempt < maxRetries; attempt++ { err := a.Send(token, title, body, data) if err == nil { return nil } if a.IsTokenInvalid(err) { // Token revoked — deactivate immediately, no retry _ = a.repo.DeactivateToken(token) return ErrTokenInvalid } // Exponential backoff: 1s, 2s, 4s wait := time.Duration(1<Sleep(wait) } return ErrMaxRetriesExceeded }

Scaling Plan

MVP

< 1k Users

  • Notification module trong Monolith
  • 1 Kafka partition/topic
  • 1 consumer goroutine
  • Cron job đơn giản
  • FCM direct call
10k Users

10k Active Users

  • Tăng lên 4–6 partitions
  • Multiple consumer instances
  • Redis idempotency layer
  • Batch FCM API (500 tokens/call)
  • Read replica PostgreSQL
50k+ Users

50k+ Active Users

  • Tách Notification Service riêng
  • gRPC internal communication
  • Dedicated Kafka cluster
  • Push provider abstraction layer
  • Horizontal pod autoscaling
